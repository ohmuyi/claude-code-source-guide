# 第17章 插件系统：从本地能力到可分发生态

## 本章要回答的问题

插件系统的目标是把“某个目录里的一组 commands、agents、skills、hooks”变成可安装、可更新、可缓存、可控策略的分发生态。它如何在不阻塞启动的前提下完成 marketplace 安装？`plugin@marketplace` 这种声明如何被解析、校验并加载为 `LoadedPlugin`？当存在 enterprise policy、缓存、ZIP、seed cache、以及 `--plugin-dir` 本地覆盖时，系统靠什么不变量维持可预期性？

## 关键源码入口

- [`src/services/plugins/PluginInstallationManager.ts`](https://github.com/ohmuyi/claude-codes/blob/main/src/services/plugins/PluginInstallationManager.ts)：`performBackgroundPluginInstallations`
- [`src/utils/plugins/pluginLoader.ts`](https://github.com/ohmuyi/claude-codes/blob/main/src/utils/plugins/pluginLoader.ts)：`loadAllPlugins`、`loadAllPluginsCacheOnly`、`clearPluginCache`
- [`src/utils/plugins/pluginLoader.ts`](https://github.com/ohmuyi/claude-codes/blob/main/src/utils/plugins/pluginLoader.ts)：`loadPluginsFromMarketplaces`、`createPluginFromPath`、`loadPluginManifest`、`mergePluginSources`

## 运行机制

安装与加载被刻意拆开。`performBackgroundPluginInstallations()` 只做 marketplace 的“物化”：它对比 `getDeclaredMarketplaces()` 与本地 `loadKnownMarketplacesConfig()` 的结果，先把缺失或 source 变化的 marketplace 标成 pending，再调用 `reconcileMarketplaces()` 在后台 clone 或 update，并把 `onProgress` 映射回 UI 状态。安装完成后分两种后续策略：若是新安装，直接 `refreshActivePlugins()` 自动刷新，修复“启动时 cache-only 加载看不到新插件”的鸡生蛋问题；若只是更新，则设置 `needsRefresh`，把应用时机交给用户通过 `/reload-plugins` 控制。

加载侧的中心是 `loadAllPlugins` 与 `loadAllPluginsCacheOnly` 的双入口。两者共享组装逻辑，差异只在 marketplace 逐项加载是否允许网络与拷贝。cache-only 模式读取 `installed_plugins.json` 记录的 `installPath`，并在 ZIP 模式下解压到 session 临时目录，保证启动不被 git clone 阻塞；full 模式则会根据 source 类型决定是否缓存，并通过 `calculatePluginVersion()` 把版本键稳定下来，必要时探测 seed cache。

`loadPluginsFromMarketplaces()` 负责把“设置声明”变成“待加载条目”。它先从 `settings.enabledPlugins` 合并 `--add-dir` 的最低优先级声明，筛出 `PluginIdSchema()` 验证通过的 `plugin@marketplace`，再预加载每个 marketplace 的 catalog，避免多个插件重复读配置。enterprise policy 的边界是“可验证来源”：`loadKnownMarketplacesConfigSafe()` 可能因配置损坏退化为空，此时若 policy 已激活，代码选择 fail-closed，直接产出 `marketplace-blocked-by-policy` 错误，而不是绕过检查继续加载。

解析到具体目录后，`createPluginFromPath()` 构造 `LoadedPlugin`：manifest 默认位于 `.claude-plugin/plugin.json`，缺失时由 `loadPluginManifest()` 生成最小 manifest 以保持可运行；组件路径既可由目录自动探测，也可由 manifest 或 marketplace entry 指定，路径检查用并行 `pathExists`，并保持错误顺序可重复。插件 settings 被严格白名单化，只允许被 schema 放行的键进入 cascade。

## 设计权衡

后台安装把“生态分发”从启动路径剥离，显著改善交互，但带来一致性问题，于是引入“新安装自动 refresh，更新仅提示 `needsRefresh`”的折中：前者解决功能缺失，后者避免用户在未知时机被动切换版本。cache-only 与 full 的双入口是典型的性能不变量设计：启动永远不做网络 I/O，而显式刷新才追求新鲜度；同时 full 加载会预热 cache-only 的 memoize，降低刷新后下游再读缓存导致的 `plugin-cache-miss`。

enterprise policy 的 fail-closed 牺牲了部分 UX，但换来策略语义不被“配置损坏”意外削弱。版本化缓存与 ZIP 或 seed cache 使得插件分发在多环境下更可控，但引入了版本键一致性的复杂性，因此代码明确避免“预克隆版本 unknown，后克隆版本变更”导致的永久重复 clone。最后，`mergePluginSources()` 选择“session 插件覆盖已安装”以优化本地开发体验，但对 managed settings 让步，保证企业管理员意图不可被 `--plugin-dir` 绕过。

## 小结

`PluginInstallationManager.ts` 把 marketplace 物化变成后台任务，`pluginLoader.ts` 把声明、策略、缓存与组件装配收敛为可重复的加载管线。双加载模式、版本化缓存、以及 fail-closed policy 构成插件生态可持续扩张的三根支柱：快启动、可治理、可分发。
