# 第12章 交互体验的最后一公里：低打扰反馈、双击拖选与卡顿自诊断

## 本章要回答的问题

1. 终端 UI 如何在“拦截鼠标键盘”与“保持原生手感”之间取得平衡？
2. 双击选词、三击选行、双击后拖选这些细节，为什么必须在 press 阶段判定？
3. 当系统卡顿或命令挂起时，如何做到自诊断但不制造新的噪声？

## 关键源码入口

- [`src/ink/components/App.tsx`](https://github.com/ohmuyi/claude-codes/blob/main/src/ink/components/App.tsx)：`processInput()`、`handleReadable()`、`handleMouseEvent()`
- [`src/tasks/LocalShellTask/LocalShellTask.tsx`](https://github.com/ohmuyi/claude-codes/blob/main/src/tasks/LocalShellTask/LocalShellTask.tsx)：stall watchdog、`looksLikePrompt()`
- [`src/tasks/LocalMainSessionTask.ts`](https://github.com/ohmuyi/claude-codes/blob/main/src/tasks/LocalMainSessionTask.ts)：后台完成通知的抑制逻辑

## 运行机制

低打扰反馈先从输入管线做起。Ink 的 `processInput()` 会把一次 `stdin` chunk 解析成多个 `ParsedInput`，并用 `reconciler.discreteUpdates(processKeysInBatch, ...)` 把整批事件包进一次离散更新，避免粘贴或连按导致的更新深度爆炸。同时 `processKeysInBatch` 只在真实用户输入，例如 key、mouse press 或 drag，时调用 `updateLastInteractionTime()`，把终端响应与 hover 噪声排除在外，避免“被动噪声”延长通知与 housekeeping 的计时。

双击拖选的核心在 `handleMouseEvent()`。它在 press 阶段用 `MULTI_CLICK_TIMEOUT_MS` 与 `MULTI_CLICK_DISTANCE` 判定 click chain，并把双击与三击直接交给 `onMultiClick(col, row, count)`，而不是等 release。这样才能让“选词高亮立刻出现”，并且支持 `double-click + drag` 的原生行为，否则会落回 char-mode selection。单击时则调用 `startSelection()` 只设置 anchor，不设置 focus，保证“点一下不高亮”，拖动后再由 `updateSelection()` 推进 focus。

链接打开同样强调“可取消”。`handleMouseEvent()` 在单击 release 后先 `onClickAt(col, row)` 做 DOM hit-test，未消费才同步解析 `getHyperlinkAt(col, row)`，但真正 `onOpenHyperlink(url)` 会被 `setTimeout(..., MULTI_CLICK_TIMEOUT_MS)` 延迟，从而允许第二次点击取消它。为避免 VS Code 或 xterm.js 自己也会开链接导致双开，代码用环境判断做了所有权划分，宁可少开，也不重复开。

卡顿自诊断分两层：UI 层与任务层。UI 层在 `handleReadable()` 里检测 `STDIN_RESUME_GAP_MS` 的长空窗，触发 `onStdinResume?.()` 以重申终端模式，覆盖 tmux attach、SSH 重连、电脑唤醒等场景；同时 catch 住 readable handler 内的异常并在必要时重新挂回监听，避免 Bun 下 stdin reader 被楔死而表现为“界面活着但不响应”。任务层则由 `LocalShellTask` 的 stall watchdog 监测输出文件增长，超过阈值后只在 tail “像交互提示”时才发出一次性通知，并且故意省略 `<status>` 标签，避免被下游当成任务终止信号。

## 设计权衡

把选择与链接逻辑搬到 Ink 层的代价是复杂度暴涨：要处理丢 release、hover 噪声、multi-click 计数、选区跨屏滚动等边界。但收益是手感可控，尤其是 press 判定带来的“即时高亮”和“选词后拖选”这类微小体验差异，在终端里往往就是是否好用的分水岭。

自诊断通知的取舍是宁静优先：stall watchdog 不追求全面识别，只在“像 prompt”时提醒，并且用一次性 latch 避免刷屏。代价是可能漏报，但它把用户注意力当作稀缺资源处理，符合 CLI 场景下“少打扰比多提醒更重要”的现实。

## 小结

最后一公里的体验来自大量看似琐碎的选择：把输入批处理为一次离散更新，把单击、双击、拖选、链接打开做成可取消的时序，把“挂起诊断”限制在真正需要人介入的时刻。它们共同指向同一件事：系统要能解释自己正在发生什么，但不能为了解释而妨碍用户继续工作。
