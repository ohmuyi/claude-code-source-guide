# 第5章 权限：权限不是一个开关，而是一条决策链

## 本章要回答的问题

1. 权限判定为什么不能简化成 allow/deny 的布尔值？
2. 一次 `tool_use` 的权限结论，来自哪些层次的规则与机制？
3. 交互式确认、hooks、classifier、swarm/worker 这些分支如何拼成一条链？

## 关键源码入口

- [`src/hooks/useCanUseTool.tsx`](https://github.com/ohmuyi/claude-codes/blob/main/src/hooks/useCanUseTool.tsx)：`useCanUseTool()`，把 `hasPermissionsToUseTool()` 的结果接到交互与多端通道
- [`src/Tool.ts`](https://github.com/ohmuyi/claude-codes/blob/main/src/Tool.ts)：`Tool.checkPermissions()`、`ToolPermissionContext`、`ToolUseContext`，定义“工具侧”和“系统侧”权限接口
- [`src/services/tools/toolExecution.ts`](https://github.com/ohmuyi/claude-codes/blob/main/src/services/tools/toolExecution.ts)：`resolveHookPermissionDecision()`、`decisionReasonToOTelSource()`，把权限结论写回遥测与执行路径
- 辅助理解：[`src/utils/permissions/permissions.ts`](https://github.com/ohmuyi/claude-codes/blob/main/src/utils/permissions/permissions.ts) 中的 `hasPermissionsToUseTool`

## 运行机制

从概念上看，权限的本质是“在给定上下文中，允许某个行为以某种方式发生”。上下文不仅有用户配置的 allow/deny/ask 规则，还有 mode，例如 `default/auto/dontAsk/bypassPermissions/plan/acceptEdits`，以及是否 headless、是否需要用户交互、hooks 和 classifier 的外部判定。这些维度叠加后，权限就不可能是一个开关，而必须是一条可解释、可回溯的决策链。

链路的第一段在 Tool 边界：每个 `Tool` 通过 `checkPermissions(input, context)` 表达“工具特定的约束”，并用 `PermissionDecisionReason` 标明原因，例如 rule、hook、mode、classifier、`safetyCheck` 等。链路的第二段在系统编排：`hasPermissionsToUseTool` 会做 mode 变换与规则匹配，例如 `bypassPermissions` 是允许，但某些 `safetyCheck` 和内容级 `ask` 规则是 bypass-immune；`dontAsk` 会把 ask 统一降级为 deny；`auto` 会优先走 `acceptEdits` 快路径，减少 classifier 调用，并在必要时把 ask 交给 classifier 决定。

链路的第三段是“如何把 ask 变成最终结论”，这由 `useCanUseTool()` 完成。它先调用 `hasPermissionsToUseTool()`，allow 直接 `ctx.buildAllow()`，deny 直接返回并记录 telemetry。遇到 ask 时，它按上下文分流：如果 `awaitAutomatedChecksBeforeDialog` 为真，先走 `handleCoordinatorPermission()`，按顺序尝试 `ctx.runHooks()` 再尝试 `ctx.tryClassifier()`，有结论就直接 resolve，避免弹窗。否则可能进入 `handleSwarmWorkerPermission()`，在 worker 侧先尝试 classifier，再把请求转发给 leader 等待回执。再否则，在 Bash 场景还会利用 `peekSpeculativeClassifierCheck()` 做一次“投机性 race”，高置信匹配可直接 auto-allow。最后才落到 `handleInteractivePermission()`，把确认请求入队，并通过 `onAllow/onReject/onAbort/recheckPermission` 等回调把用户操作转成 `PermissionDecision`，同时可持久化 permission updates。

执行侧还会把这条链的结果“落账”。在 `toolExecution.ts` 里，非交互路径会通过 `decisionReasonToOTelSource()` 归一化来源，例如 config、hook、`user_permanent`、`user_temporary`、`user_reject`，并在需要时补发 `tool_decision` 事件，保证 headless 与 interactive 的观测语义一致。

## 设计权衡

决策链的好处是可扩展和可解释，代价是复杂与延迟。复杂主要来自分支爆炸，源码用“分层决策”抑制它：Tool 只提供局部规则，系统集中处理 mode、rules、headless、classifier，交互层只负责把 ask 变成一次可完成的事务。

延迟主要来自自动化检查与外部服务。这里的取舍很明显：coordinator 模式选择先 await 自动化检查再弹窗，减少无谓打断；普通交互模式则允许“用户先动起来”，并用 `onUserInteraction` 去取消 classifier 的指示与自动批准窗口。与此同时，fail-closed 的策略也在代码里很直白，例如 headless 且必须人工确认的 `safetyCheck` 会直接 deny，避免后台 agent 卡死等待。

## 小结

Claude Code 的权限系统把“规则匹配、模式转换、自动化检查、交互确认、多端同步、遥测归因”串成一条链。你看到的不是一个权限开关，而是一套让不同运行形态，例如 REPL、coordinator、swarm worker、headless，共享同一语义的决策引擎。
