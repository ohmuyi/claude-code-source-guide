# 第15章 记忆系统：把当前对话沉淀为未来上下文

## 本章要回答的问题

1. “记忆”在 Claude Code 里为什么必须是文件，而不是隐式状态或数据库？
2. `MEMORY.md` 为什么被设计成索引而不是正文，它如何控制上下文体积与可维护性？
3. 自动记忆、团队记忆与长会话日志模式如何共存，又如何守住安全边界？

## 关键源码入口

- [`src/memdir/memdir.ts`](https://github.com/ohmuyi/claude-codes/blob/main/src/memdir/memdir.ts)
- `loadMemoryPrompt(...)`、`buildMemoryLines(...)`、`buildMemoryPrompt(...)`
- `truncateEntrypointContent(...)`、`ensureMemoryDirExists(...)`
- 关联路径与开关：[`src/memdir/paths.ts`](https://github.com/ohmuyi/claude-codes/blob/main/src/memdir/paths.ts)

## 运行机制

概念上，这套系统把“未来要用的信息”从对话流里抽离出来，落到可审计、可编辑、可版本化的 Markdown 文件中。它强调的是长期协作的稳定接口：用户可以看到记忆写进了哪里，什么被记录、什么不该记录都有明确规则，模型也不需要靠隐式状态去猜测“上次你说过什么”。

机制上，`memdir.ts` 的中心是 prompt 构建与注入。`buildMemoryLines(...)` 生成一段“行为规范”，把可保存内容限制在封闭的类型体系里，并明确排除“可从项目状态推导的信息”。当启用索引模式时，它把保存流程拆成两步：先为每条记忆写独立文件，再在 `ENTRYPOINT_NAME`，也就是 `MEMORY.md`，里加一行指针，避免把大段正文塞进永远会被加载的入口文件。`buildMemoryPrompt(...)` 会同步读取 `MEMORY.md`，并通过 `truncateEntrypointContent(...)` 同时施加行数与字节上限，必要时追加清晰的 WARNING，告诉模型与用户“只加载了部分”，从源头限制上下文膨胀与长行异常。

加载策略由 `loadMemoryPrompt(...)` 决定，它根据 feature gate 与运行态选择多种形态：普通 auto memory、auto 和 team memory 的组合提示，或在长生命周期 assistant 模式下改用按日期 append-only 的 daily log。为减少模型在写入前浪费回合，源码提供 `ensureMemoryDirExists(...)` 并在提示里用存在性文案明确“目录已存在，直接 Write”。若启用 past context 搜索能力，还会根据是否有 embedded tools 或 REPL 模式选择命令或工具形态，给出可执行的检索指令。

路径与安全边界在 `paths.ts` 被进一步固化：`isAutoMemoryEnabled()` 有清晰的优先级链，`validateMemoryPath(...)` 对路径做绝对路径、近根目录、UNC、空字节等校验，并且刻意不信任 projectSettings 的 `autoMemoryDirectory`，避免恶意仓库把记忆目录指向敏感路径，从“可写目录”变成“权限逃逸”。

## 设计权衡

第一组权衡是“可用性”对“可控性”的取舍。让 `MEMORY.md` 永远加载能显著提升连续性，但必须配套截断策略与“索引而非正文”的结构，否则记忆越积越多就会反噬上下文。daily log 模式把写入简化为追加，也降低了维护成本，但它依赖后续蒸馏流程把日志变成可检索的主题文件与索引，实时可读性会弱一些。

第二组权衡是“持久化能力”与“安全边界”的对抗。记忆系统本质上扩大了模型的文件写入价值，因此源码在开关、路径来源与校验上做了多层防护，并通过 telemetry 记录禁用原因与加载情况。代价是配置链更复杂，且在 remote 或非交互环境下会因存储条件不足而主动关闭，换取整体系统的不出界。

## 小结

`memdir.ts` 把“记忆”实现为一套可治理的文件协议：用 `MEMORY.md` 做稳定入口与索引，用独立文件承载正文，用截断控制体积，用路径校验守住边界，再用加载策略把 auto、team 与日志形态组合起来。它真正沉淀的不是内容本身，而是让内容在未来仍然可被安全、稳定地取用的结构。
