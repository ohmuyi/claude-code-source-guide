# 第18章 压缩与演进：在有限上下文里维持长期运行

## 本章要回答的问题

长会话的敌人不是“信息太多”，而是“上下文预算固定”。压缩机制要回答：哪些消息该被总结，哪些状态必须保留？如果压缩请求本身也触发 `prompt-too-long`，系统如何自救而不是让用户卡死？压缩后的第一回合，模型需要哪些最小恢复信息，才能延续先前的工作流而不反复 `Read`、不丢计划、也不忘已调用的 skills？

## 关键源码入口

- [`src/services/compact/compact.ts`](https://github.com/ohmuyi/claude-codes/blob/main/src/services/compact/compact.ts)：`compactConversation`、`partialCompactConversation`
- [`src/services/compact/compact.ts`](https://github.com/ohmuyi/claude-codes/blob/main/src/services/compact/compact.ts)：`streamCompactSummary`、`truncateHeadForPTLRetry`
- [`src/services/compact/compact.ts`](https://github.com/ohmuyi/claude-codes/blob/main/src/services/compact/compact.ts)：`createPostCompactFileAttachments`、`createSkillAttachmentIfNeeded`
- [`src/services/compact/compact.ts`](https://github.com/ohmuyi/claude-codes/blob/main/src/services/compact/compact.ts)：各类 post-compact token budget 常量

## 运行机制

`compactConversation()` 的主流程是“先生成摘要，再重建运行态”。它先计算压缩前 token 数，执行 `executePreCompactHooks()` 并合并用户与 hook 的自定义指令，然后用 `getCompactPrompt()` 生成摘要请求。摘要生成走 `streamCompactSummary()`，优先尝试 `runForkedAgent()` 复用主对话的 prompt cache 前缀，失败则回退到普通流式路径，并显式禁用 thinking，确保压缩 agent 只产出文本摘要。

关键的边界条件是：压缩请求本身也可能触发 `PROMPT_TOO_LONG_ERROR_MESSAGE`。代码在循环里识别该前缀，调用 `truncateHeadForPTLRetry()` 按 API round 分组丢弃最老的若干组消息后重试，最多若干次。这是一种“亏损但可恢复”的逃生通道，优先保证会话继续运行，而不是追求完美摘要。

摘要成功后，系统清空 `readFileState` 与 nested memory 路径缓存，然后并行生成 post-compact 附件。`createPostCompactFileAttachments()` 从最近读过的文件里选有限数量的文件，用 `FileReadTool` 重新读取并限制单文件与总预算，避免“恢复态”本身把上下文吃爆。计划与模式也被显式恢复：若存在 plan 文件则追加引用，若当前处于 plan mode 则追加状态说明。skills 侧通过 `createSkillAttachmentIfNeeded()` 注入已调用 skills：按最近调用排序，每个 skill 头部截断并受总预算限制，确保“用过的规则”在压缩后仍可见，但不会无限膨胀。

最后，压缩会补齐“被摘要吃掉的 delta”：重新生成 deferred tools、agent listing、MCP instructions 等差量附件，并插入 boundary message 与摘要消息，计算压缩后的真实 token 数作为下一回合是否会再次触发自动压缩的信号。压缩前后还会执行 session start hooks 与 post-compact hooks，让外部扩展有机会参与这一生命周期。

## 设计权衡

压缩必然是有损的，因此设计重点从“保留所有内容”转向“保留可继续工作的最小闭包”：最近文件、plan、plan mode、已调用 skills、工具与 MCP 指令差量。`createPostCompactFileAttachments()` 选择“重新 Read 而不是复用旧内容”，牺牲一点延迟，换来内容与权限校验路径一致，避免把过期或未验证的文本当作真相。对 skills，系统选择“每个 skill 截断保头”而不是“超预算就整段丢弃”，这是对提示工程实践的贴近。

prompt cache sharing 能显著降低成本，但对参数一致性非常敏感，因此 fork 路径刻意不设置某些输出参数，避免 thinking 配置差异导致 cache 失效；同时提供 streaming fallback 和重试开关，宁可慢也要稳。PTL 自救通过丢弃最老 API rounds 保证“不会卡死”，但它把损失集中在早期上下文，这要求后续附件恢复机制足够强，能把工作态重新拼回来。

## 小结

`compact.ts` 把压缩从“摘要一下聊天记录”升级为“有预算、有生命周期、有恢复态的演进机制”。它用 hooks 扩展边界，用 cache sharing 与 PTL 重试处理性能与极端失败，用文件与 skills 的预算化重注入维持长跑会话的连续性。最终目的不是记住一切，而是在有限上下文里持续产生可用的下一步。
