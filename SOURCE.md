# 源码说明

这本书解读的对象是 Claude Code 的源码目录，但源码仓库与书稿仓库分开维护。

## 当前对应源码

当前公开源码地址是：

- `https://github.com/ohmuyi/claude-codes/tree/main/src`

如果你在本地配套阅读，建议把源码单独 clone 到任意工作目录，例如：

- `<workspace>/claude-code`

## 为什么不把源码直接放进本仓库

不建议把源码目录直接拷贝或移动进书稿仓库，原因很简单：

- 目录会变得臃肿，读者很难一眼分清哪些是书稿、哪些是源码
- 仓库搜索结果会混入大量源文件，降低阅读与导航效率
- 后续更新时，书稿和源码的 diff 会缠在一起，不利于维护
- 如果源码本身来自其他项目，还会带来来源、版本和许可边界不清的问题

因此，这里采用更稳妥的方式：

- 书稿仓库只放书稿、目录、贡献说明和协作模板
- 源码仓库单独放置，按自己的节奏更新
- 书稿通过章节中的源码路径说明和外部链接指向对应源码

## 章节中的源码路径写法

为了方便 GitHub 读者阅读，章节中不使用作者本机的绝对路径，而统一使用源码仓库内的相对路径，例如：

- `src/main.tsx`
- `src/query.ts`
- `src/services/mcp/client.ts`

这种写法的好处是：

- 不依赖作者本地机器目录
- 读者在自己的 clone 中更容易定位
- 以后源码目录位置变化时，书稿不需要跟着改本机路径

## 推荐的本地目录结构

推荐把两者并列放在同一级目录下：

```text
<workspace>/
├── claude-code-source-guide/
└── claude-code/
```

这样做的好处是：

- 本地查阅方便
- 仓库边界清晰
- 后续分别提交、迁移、备份都更简单

## 公开发布前建议补充的信息

如果以后要把这本书公开发布到 GitHub，建议在这个文件里继续补充：

1. 被解读源码的公开仓库地址
2. 对应的 commit、tag 或发布日期
3. 源码本身的许可证信息
4. 本书引用源码时采用的版本基线

这样读者在阅读某一章时，就能明确知道“这章讲的是哪一版源码”，不会把不同版本的实现混在一起。
